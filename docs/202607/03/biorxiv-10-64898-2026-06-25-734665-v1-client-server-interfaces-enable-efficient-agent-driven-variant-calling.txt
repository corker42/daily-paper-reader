Abstract 6

Background: Large language model (LLM) agents increasingly automate bioinformatics 7analyses, but most existing bioinformatics tools were built for standalone use by human experts. 8An agent driving such a tool must reason about its installation, configuration, and execution from 9documentation for human, spending many turns, tokens, and tool calls per result. How a method 10 is exposed to an agent can therefore matter as much as the method itself. By designing agentic 11 interfaces for these tools, agent can reduce such overhead and improve the reliability of agent-12 driven analyses. 13 

Findings: To test this design , we re-architected Clair3, a widely used deep-learning-based long-14 read variant caller, into a client-server system, Clair3-Connect. The client performs all genomics 15 related processing and holds the identifiable data. The server runs only neural-network inference, 16 and the client sends only feature tensors to the server, while sample identifiers and genomic 17 context remain on the client. The client exposes schema-defined agent-facing tools that an agent 18 invokes through single structured calls. On an APOE diplotyping task, all 60 agent runs were 19 correct. The agentic tools used 12K tokens in 3 turns, 6.8 to 14 times fewer tokens than the shell-20 driven baselines (81K-163K tokens), at about a quarter the wall-clock time and far more stably 21 (4% versus 35% token usage variation). Dropping the pileup and phasing stages to keep the 22 client light left SNP F1 within 0.1-0.3 points of standard Clair3 by 50× coverage, while mutual 23 TLS and AES-256-GCM encryption added 7.2% to end-to-end runtime. 24 

Conclusions: Recasting an established algorithm as developer-built, agentic tools behind a 25 secure client-server boundary makes it more efficient, reliable, and easier to deploy for an LLM 26 agent than a third-party wrapper, which cannot recover the defaults and conventions only its 27 developers know. Agentic interfaces should be a first-class deliverable of bioinformatics tool 28 development. 29

The wider software ecosystem is moving toward standardized client-server tool interfaces, 59 including protocols such as the Model Context Protocol [18]. In biology, early evidence from 60 agent-designed database access suggests that replacing human-facing interfaces with 61 deterministic, agent-facing ones can improve agent accuracy and reproducibility [19]. 62 Bioinformatics algorithms, however, are still most commonly delivered as standalone tools 63 rather than as semantically explicit services. 64 Moreover, genomics also imposes constraints that make this interface problem more than a 65 software-engineering issue. Sequencing data can contain sensitive or potentially identifying 66 information, and large neural-network-based workflows may require hardware or runtime 67 dependencies that are not available on every client machine. Moving full sequencing files to a 68 remote environment can be undesirable, but local execution can be expensive or difficult to 69 standardize. An agent-facing genomics system therefore needs to address three requirements at 70 once: it must expose the algorithm in a form that agents can use reliably, keep sample-specific 71 context under client control, and allow compute-intensive inference to be shared when 72 appropriate. 73 To examine this design in a mature genomics tool, we use Clair3 as a representative example and 74 re-architect it into Clair3-Connect, a client-server system for agent-facing variant calling. Clair3-75 Connect keeps sample-specific genomic processing on the client and exposes neural-network 76 inference as a stateless server-side service. The agent interacts with the caller through structured 77 tool calls rather than by scripting a command-line program. We evaluate this design by 78 comparing agentic tool access with shell-based agent access, measuring the cost and variability 79 of agent-driven variant calling, quantifying the accuracy trade-off introduced by a lightweight 80 client, and estimating the overhead of secure transport. Findings 85 

Overview of the system 86 Clair3-Connect re-architects Clair3 around a single computational boundary: sample-specific 87 genomic processing remains on the client, whereas neural-network inference is exposed as a 88 stateless server-side service (Figure 1). The client reads the alignments and reference, selects 89 variant candidate sites, constructs Clair3 feature tensors for these sites, and decodes returned 90 probabilities into variant calls. The server runs only the neural network, computing tensors to 91 per-site probabilities without receiving the surrounding genomic or file-system context. Thus, the 92 exchanged data are limited to feature tensors and request identifiers than do not encode sample, 93 genomic or file system context. This keeps sample-specific context on the client and reduces 94 exposure of sensitive or potentially identifying information when inference is performed on 95 shared or remote hardware. 96 The same boundary separates GPU-intensive inference from the client-side genomics workflow. 97 No local GPU is required for the client side, and the server can batch and accelerate inference 98 across requests. The structured client interface also makes Clair3-Connect directly callable by 99 applications and LLM agents. Instead of scripting a command-line program and parsing 100 unstructured files, an agent invokes structured tool calls with defined inputs, outputs and error 101 states. The following sections evaluate the effect of this design on agent cost, variant-calling 102 accuracy, and secure transport overhead. Transport, runtime, and interface details are given in 103 the Methods section. 104

Driving Clair3-Connect from an LLM agent 105 We compared three ways of exposing Clair3 to an LLM agent while holding the model and task 106 fixed (Methods). The RawBinary condition only gave the agent shell access to the stock Clair3 107 program [17]. The Skills condition added a skill document provided by Clair3’s developers to 108 the same shell environment. The Tools condition exposed only the Clair3-Connect agentic tool 109 interface. Each condition was run in ten independent repeats per sample. All 60 runs returned the 110 correct APOE diplotype; only the cost and predictability differed (Figure 2). The prompts and 111 per-repeat metrics are reported in Supplementary Note S1, with one representative transcript in 112 Supplementary Note S4. 113 The way Clair3 is exposed affected the cost of driving it, with the Tools condition interface 114 being the most efficient. The RawBinary condition used 81K tokens over 10 turns on average, 115 and Skills used 163K tokens over 7.5 turns. The Tools condition reached the answer in 12K 116 tokens and 3 turns, corresponding to 6.8-fold fewer tokens than the RawBinary condition and 14-117 fold fewer tokens than the Skills condition, with approximately one quarter of the wall-clock 118 time. The Skills condition was most expensive because the skill document was re-injected on 119 every turn. In both shell-based settings, the agent also had to inspect bulky unstructured outputs, 120 including BAM headers, region dumps, command-line logs, and VCF text. 121 The agentic tool interface also reduced run-to-run variability. Token usage varied by only 4% 122 across Tools repeats, compared with approximately 35% in both shell-based conditions (Figure 123 2). Agentic tool calls return compact, task-specific results and restrict the agent to a small action 124 space. Shell access, by contrast, leaves the agent to decide which commands to run, which 125 outputs to inspect, and how much intermediate evidence to re-check. Thus, the interface affects 126 not only the cost of an agentic workflow, but also its run-to-run stability. 127

The HG00136 sample illustrates why agent-facing tools should encode algorithm-specific 128 semantics. HG00136 is ε4/ ε4, and the rs7412 site is homozygous for the reference allele. Under 129 Clair3’s default behavior, such sites are absent from the VCF unless reference calls are requested 130 with --print_ref_calls . None of the shell-based runs used this option, and the skill 131 document did not mention it. The shell arms reached the correct answer only because the prompt 132 supplied a benchmark-specific convention: for the two predefined APOE loci, an absent record 133 in Clair3’s variant-only VCF should be treated as reference for the purpose of deriving the 134 diplotype. This was not intended as a general rule for VCF interpretation. In general, the absence 135 of a VCF record should not be interpreted as evidence for a homozygous-reference genotype, 136 because missing records may also reflect low confidence, insufficient coverage, filtering, or non-137 callable regions. 138 The Tools arm did not require this convention in the prompt. The developer-built genotype query 139 returned an explicit 0/0 genotype for positions absent from the VCF (Supplementary Note S4). 140 This example shows a limitation of directly wrapping command line tools for agentic use: 141 defaults, flags, and missing-record conventions are often part of the algorithm’s operational 142 semantics, but they are not reliably recoverable from final output files alone. Exposing them 143 through typed tools allows the agent to use the caller with less parsing, less redundant 144 verification, and fewer opportunities for unsupported assumptions. 145 Together, these results show that a client-server typed-tool interface improves both the efficiency 146 and robustness of agent-driven variant calling. They also suggest that agent-facing infrastructure 147 should be treated as part of bioinformatics algorithm development, not as a thin wrapper added 148 after the fact. 149

System performance and implementation trade-offs 150 To minimize client-side computation, Clair3-Connect deviates from the standard Clair3 pipeline. 151 Standard Clair3 first applies a pileup model to all candidate sites, phases and haplotags the reads 152 with WhatsHap[20] or LongPhase[21], then runs the full-alignment model on the remaining sites 153 and merges the outputs. Clair3-Connect builds only the full-alignment tensor and calls from it 154 directly, removing the pileup and phasing stages. 155 The accuracy gap introduced by removing pileup and phasing was concentrated at low coverage 156 and in noisier read data and narrowed to a small residual difference at higher coverage or with 157 higher-accuracy reads. For SNPs on chromosome 20, the full-alignment-only configuration 158 achieved SNP F1 scores 9.5 and 12.5 percentage points lower than standard Clair3 on Illumina 159 and ONT data, respectively (Figure 3). The ONT accuracy gap is driven by precision, which falls 160 to 69.5% from 90.1% as the model retains more false positives where reads are sparse. The gap 161 then narrows with depth, falling below one F1 point by 30× and to 0.1-0.3 points by 50× on both 162 platforms. On PacBio HiFi it stays below 0.1 points at every depth, because per-read accuracy 163 reducing the reliance on phasing. Indels show the same pattern with larger, more persistent gaps, 164 because indel calling is more sensitive to read-level alignment noise and benefits more from 165 phasing and haplotagging. The residual gap is largest on ONT, where the full-alignment-only 166 client loses about 8 F1 points at 10× and still 4 F1 points by 50× (Figure 3, bottom row). The 167 complete benchmark for both chromosomes is in Supplementary Note S2. To conclude, at 168 moderate-to-high depths typical of many whole-genome sequencing applications, Clair3-169 Connect retains most of standard Clair3’s accuracy while reducing the computation that must be 170 performed locally. 171

Transport security adds measurable but modest end-to-end overhead 175 A shared or remote inference server raises the question of data exposure in transit. Clair3-176 Connect addresses this risk at two levels. First, the client-server boundary limits what can leave 177 the client. The server receives feature tensors, but not explicit sample identifiers, file paths, 178 genomic coordinates, or other contextual metadata. Second, Clair3-Connect can protect the 179 transmitted tensors with two standard mechanisms. Mutual TLS (mTLS) authenticates both 180 endpoints and encrypts the channel. Per-payload AES-256-GCM encryption (AEAD) encrypts 181 each tensor and binds it to its model, shape, and content. These mechanisms provide standard 182 channel confidentiality, endpoint authentication, and payload integrity. Here, we measure the 183 cost of applying them to Clair3-Connect’s tensor-based workflow. 184 We first measured the transport layers in isolation. The full-alignment tensor used in this 185 benchmark had low byte-level entropy (0.82 bits/byte) and was highly compressible: ZSTD-1 186 reduced the 752 KB ONT tensor to ~22 KB, corresponding to a 34.0× reduction and ~3% of the 187 original size. This benchmark used the full 752 KB tensor without compression and made 188 inference the only server-side work, placing the tensor on the request critical path. Under 189 plaintext transport, per-request latency was 13.4 ms. Latency increased to 20.8 ms with AEAD 190 and to 21.6 ms with mTLS+AEAD (Figure 4a). The absolute AEAD cost was stable at 191 approximately 7.4 ms. 192 The deployment-relevant cost was smaller in the full workflow. In a full-alignment-only run, 193 client-side candidate scanning dominated runtime and absorbed much of the transport overhead. 194

Discussion 199 In this study, we use Clair3-Connect to show that the design of a bioinformatics tool interface 200 can strongly affect the efficiency and reliability of LLM-agent tool use. We demonstrated that 201 the agentic interface made agent-driven variant calling more efficient and more stable than shell-202 based access, not by changing the biological task or the underlying model, but by changing what 203 the agent could see and how it could act. This suggests that agent-facing interfaces should be 204 treated as part of bioinformatics tool development, not as optional add-ons developed after the 205 tool is released. 206 The need for such interfaces is especially clear in mature bioinformatics tools. Correct use often 207 depends on defaults, flags, and output conventions that are familiar to developers but difficult to 208 infer from final files alone. A command-line wrapper can expose the executable, but it cannot 209 reliably by itself expose the algorithm’s operational semantics. Agentic tools allow developers to 210 make these operational details explicit, including structured outputs, explicit defaults, error states, 211 version information, and provenance. For agent-driven workflows, this reduces the need for 212 fragile parsing and unsupported interpretation of logs, headers, and intermediate files. 213 Clair3-Connect also reflects the constraints of adapting existing standalone tools to this mode of 214 use. Most bioinformatics software was designed for expert users and file-based workflows, not 215 for agent orchestration through typed calls. In Clair3-Connect, keeping the client lightweight 216 required simplifying the standard Clair3 workflow, and this introduced accuracy costs in the 217

Methods 229 

System architecture and inference workflow 230 Clair3-Connect is implemented in Rust and comprises three components, a client that performs 231 the genomic stages of variant calling, a stateless server that runs the neural network, and the 232 gRPC transport layer between them. The client takes an indexed BAM or CRAM alignment, the 233 reference genome, and the regions to call, supplied as a single interval, a BED file, or a list of 234 intervals. Within each region it selects candidate sites using Clair3’s default allele-frequency 235 criteria, with minimum allele frequency of 0.08 for SNPs and 0.15 for indels at a minimum read 236 depth of 2. We disabled the pileup model and the read phasing module, so the tensor’s haplotype 237 channel in the resulting tensor carries no haplotype assignment. For every candidate the client 238 builds the full-alignment feature tensor defined by Clair3, an int8 tensor with a per-platform read 239 depth of 89 rows for Oxford Nanopore and 55 for PacBio HiFi and Illumina, across 33 flanking 240

The 242 client sends the candidate tensors to the server in batches of 32, receives the per-site genotype 243 probabilities, and decodes them into SNP and indel calls using Clair3’s output logic. It writes the 244 calls to a Clair3-compatible VCF as a self-contained command-line caller or exposes them to an 245 application or LLM agent through the typed tools described below. 246 The server performs only inference. It loads one or more Clair3 models as TorchScript modules 247 using libtorch, on CPU or a specified GPU, and processes each request by running the selected 248 model on a batch of feature tensors and returning the per-site probabilities. A dynamic batcher 249 coalesces concurrent requestsuntil either 64 tensors are available or 4 ms has elapsed, so one 250 server can accelerate inference for many clients. The server is stateless across requests apart 251 from the loaded models. Its request schema has no fields for file paths, genomic coordinates, or 252 sample or read names. Transport security 319 All client-server communication uses gRPC, secured by two independent layers that are each 320 optional. Mutual TLS encrypts the channel and requires both endpoints to present a certificate 321 chained to a configured authority. This gives channel confidentiality and integrity and admits 322 only authenticated clients. Before payload encryption, tensor payloads can optionally be 323 compressed with Zstandard (ZSTD).The second layer applies authenticated encryption to each 324 tensor payload, independently of the channel. Each tensor is sealed with AES-256-GCM under a 325 fresh 96-bit random nonce and a 16-byte authentication tag. The additional authenticated data 326 covers the request’s model identifier, data type, tensor shape, and payload hash. This binds each 327 ciphertext to these request metadata fields and prevents it from being accepted into a valid tensor 328 under a different model or shape. Because the seal is end-to-end, a TLS-terminating intermediary 329 that decrypts the channel still sees only the encrypted payload. Key management, including 330 nonce limits and rotation, is documented with the released source code. 331

All measurements used sample HG003 on Oxford 337 Nanopore chromosome 20, on a 128-core host with one RTX 4090, over loopback, so the 338 reported costs are pure compute and exclude external network latency. The per-request payload 339 was an int8 full-alignment tensor ( ≈752 KB). 340 Cost was measured in two settings. The isolated cost used a pure-inference generator that 341 performs no candidate scan, placing the full payload encryption on the request critical path; for it 342 we report per-request latency. The end-to-end cost was the wall-clock time of a real full-343 alignment-only run over chromosome 20:1-8 Mb, which is bound by the CPU candidate scan. 344 We also verified the layers’ guarantees. For confidentiality we confirmed on a real full-alignment 345 tensor that the AES-256-GCM ciphertext showed no detectable byte-level structure of the 346 plaintext under our tests. Authentication was tested by presenting a valid client certificate, no 347 certificate, and a certificate from an unrelated authority. Integrity was tested with a transport 348 unit-test suite that tampers with the ciphertext, the additional authenticated data, the key, and the 349 nonce length. The full statistics and outcomes are in Supplementary Note S3. 350
